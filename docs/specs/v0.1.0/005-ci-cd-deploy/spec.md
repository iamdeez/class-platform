## 배경 및 목적

001~004에서 강의·수강신청·커뮤니티·통계 기능을 모두 구현했지만, 지금까지는 로컬 `docker-compose` 환경에서만 실행·검증할 수 있었다. 채용공고가 요구하는 "CI/CD 파이프라인 구축·운영" 역량을 실제로 보여주고, 포트폴리오 심사자가 로컬 세팅 없이 접속해 볼 수 있는 배포 환경을 마련하는 것이 이 spec의 목적이다.

연습 프로젝트 특성상 실제 트래픽·예산이 없으므로, 비용 없이(무료 티어) CI 자동화와 배포를 구성하는 것을 전제로 한다. 이 프로젝트는 MySQL·MongoDB·Redis 세 개의 저장소를 동시에 사용하므로, 배포 환경도 이 세 저장소 각각을 감당할 수 있어야 한다.

## 사용자 스토리 (User Stories)

- **US-001**: 이 프로젝트를 검토하는 사람(채용 담당자 등)으로서, 로컬 환경을 세팅하지 않고도 실행 중인 데모에 바로 접속해 보고 싶다.
- **US-002**: 개발자(나)로서, 코드를 push하거나 PR을 올리면 자동으로 빌드·테스트가 실행되어 회귀를 병합 전에 발견하고 싶다.
- **US-003**: 개발자(나)로서, main 브랜치에 테스트를 통과한 코드가 병합되면 수동 배포 작업 없이 자동으로 최신 버전이 반영되길 원한다.
- **US-004**: 개발자(나)로서, 테스트가 실패한 코드는 어떤 경우에도 배포되지 않기를 원한다.

## 기능 요구사항 (Functional Requirements)

- **FR-001**: 코드가 push되거나 PR이 제출되면 자동으로 전체 테스트 스위트가 실행된다.
- **FR-002**: 테스트가 실패한 PR은 병합이 차단된다.
- **FR-003**: main 브랜치에 테스트를 통과한 코드가 병합되면 자동으로 배포 환경에 반영된다.
- **FR-004**: 테스트가 실패한 커밋은 배포 단계로 진행되지 않는다.
- **FR-005**: 배포된 애플리케이션은 공개적으로 접근 가능한 URL로 서비스된다.
- **FR-006**: 애플리케이션 실행에 필요한 민감 정보(DB 자격증명, API 키 등)는 소스 코드나 버전 관리 대상 설정 파일에 평문으로 포함되지 않는다.

## 비기능 요구사항 (Non-Functional Requirements)

- **NFR-001**: CI 파이프라인의 빌드·테스트 실행은 무료 할당량 내에서 처리되어 비용이 발생하지 않는다.
- **NFR-002**: 배포 환경(컴퓨트·MySQL·MongoDB·Redis)은 모두 무료 티어로 운영되어 비용이 발생하지 않는다.

## 수용 기준 (Acceptance Criteria)

- **SC-001** (`FR-001`): 코드를 push하거나 PR을 생성하면, 별도 수동 트리거 없이 CI에서 전체 테스트 스위트(단위·슬라이스·통합 테스트 전체)가 자동 실행된다.
- **SC-002** (`FR-002`): 테스트가 실패하는 커밋을 담은 PR은 GitHub 병합 버튼이 비활성화된다(필수 상태 검사 실패로 병합 차단).
- **SC-003** (`FR-003`): main 브랜치에 테스트를 통과한 커밋이 병합되면, 병합 완료 시점으로부터 15분 이내에 배포 환경에서 새 버전의 응답을 확인할 수 있다.
- **SC-004** (`FR-004`): 테스트를 의도적으로 실패시키는 커밋이 main에 병합되면(예: 관리자 강제 병합), 배포 워크플로우가 실행되지 않거나 실행 중 실패로 종료되어 이전 버전이 유지된다.
- **SC-005** (`FR-005`): 배포 완료 후 공개 URL로 헬스체크 요청을 보내면 200 응답을 받는다.
- **SC-006** (`FR-006`): 저장소(git history 포함)의 어떤 파일에도 실제 DB 비밀번호·API 키가 평문으로 존재하지 않는다. 모든 민감 정보는 CI/배포 플랫폼의 비밀 저장소(secrets)를 통해 주입된다.

## 범위 외 (Out of Scope)

- **테스트 커버리지 확대** — 이 spec은 기존 테스트 스위트를 CI에 자동화하는 것이며, 새로운 테스트 케이스를 추가하는 것은 범위가 아니다.
- **오토스케일링·로드밸런싱** — 무료 티어 단일 인스턴스로 충분하다(포트폴리오 목적, 실제 트래픽 없음).
- **Blue-Green·카나리 등 고급 배포 전략** — 단순 배포(전체 교체)만 지원한다.
- **모니터링·알림 체계(APM, 에러 트래킹 등)** — `infra.md`에 별도 spec 후보로 남아 있으며, 이번 spec 범위가 아니다.
- **커스텀 도메인 연결** — 관리형 서비스가 기본 제공하는 서브도메인을 사용한다.
- **데이터베이스 백업·재해복구 전략** — 무료 티어 특성상 별도로 설계하지 않는다(포트폴리오 데이터 유실을 허용한다).
- **Kubernetes 등 컨테이너 오케스트레이션 도입** — 단일 컨테이너 배포로 충분하다.

## 미결 사항 (Open Questions)
