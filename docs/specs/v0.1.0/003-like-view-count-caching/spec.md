## 배경 및 목적

001, 002 spec의 로드맵에서 예고된 3단계에 해당한다. 002까지 구현한 게시글에 좋아요·조회수 집계 기능을 추가하고, 이 집계를 캐시(Redis)로 처리해 채용공고의 "대용량 콘텐츠 트래픽" 대응 역량을 연습한다.

좋아요·조회수는 게시글 조회마다(혹은 매 좋아요 클릭마다) 값이 바뀌는 쓰기 집약적(write-heavy) 데이터다. 이를 매번 영속 저장소(MongoDB)에 직접 반영하면 동시 요청이 몰릴 때 쓰기 부하가 커진다. 캐시에서 원자적으로 집계한 뒤 주기적으로 영속화하는 구조로, 카운트 유실 없이 부하를 낮추는 설계를 연습하는 것이 이 spec의 핵심이다.

## 사용자 스토리 (User Stories)

- **US-001**: 사용자로서, 마음에 드는 게시글에 좋아요를 눌러 반응을 표현하고 싶다.
- **US-002**: 사용자로서, 실수로 누른 좋아요를 취소하고 싶다.
- **US-003**: 사용자로서, 게시글의 조회수를 보고 얼마나 많은 사람이 읽었는지 가늠하고 싶다.
- **US-004**: 사용자로서, 인기 게시글 목록을 보고 지금 화제가 되는 글을 빠르게 찾고 싶다.

## 기능 요구사항 (Functional Requirements)

- **FR-001**: 사용자는 게시글에 좋아요를 누를 수 있다.
- **FR-002**: 사용자는 이미 누른 좋아요를 취소할 수 있다 (토글).
- **FR-003**: 동일 사용자가 동일 게시글에 좋아요를 여러 번 요청해도 실제 반영되는 좋아요는 최대 1개다 (중복 반영 방지).
- **FR-004**: 게시글 조회 시 현재 좋아요 수가 함께 제공된다.
- **FR-005**: 게시글을 조회할 때마다 조회수가 1 증가한다 (동일 사용자의 중복 조회도 매번 증가 — 중복 방지 없음).
- **FR-006**: 게시글 조회 시 현재 조회수가 함께 제공된다.
- **FR-007**: 사용자는 좋아요 수 기준 내림차순 상위 10개의 인기 게시글 목록을 조회할 수 있다.

## 비기능 요구사항 (Non-Functional Requirements)

- **NFR-001**: 좋아요·조회수 카운트 증가/감소 연산은 캐시(Redis)의 원자적 연산으로 처리하여, 동시 요청이 몰려도 카운트가 유실되지 않아야 한다.
- **NFR-002**: 캐시(Redis) 연결 장애가 발생해도 게시글 조회 자체는 실패하지 않아야 한다 (장애 격리 — 002의 AI 처리 장애 격리 원칙과 동일한 사상).
- **NFR-003**: 인기 게시글 목록 조회는 매 요청마다 전체 게시글을 정렬하지 않고, 캐시에 유지되는 랭킹 자료구조를 사용한다.
- **NFR-004**: 좋아요·조회수는 캐시에서 즉시 반영되고, 영속 저장소(MongoDB)에는 주기적으로 동기화된다 (실시간 동기 반영을 요구하지 않는다).
- **NFR-005**: 게시글 상세 조회는 캐시(Redis)를 우선 사용하여 응답하며, 캐시 미스 시에만 영속 저장소(MongoDB)를 조회한다 (조회 성능 최적화 — 반복 조회되는 게시글의 DB 접근을 줄인다).

## 수용 기준 (Acceptance Criteria)

- **SC-001** (`FR-001`, `FR-003`): 동일 사용자가 같은 게시글에 좋아요를 두 번 요청해도 좋아요 수는 1만 증가한다.
- **SC-002** (`FR-002`): 좋아요를 누른 뒤 취소하면 좋아요 수가 좋아요를 누르기 전 값으로 돌아온다.
- **SC-003** (`FR-004`): 게시글 상세 조회 응답에 현재 좋아요 수가 포함된다.
- **SC-004** (`FR-005`): 동일 게시글을 3회 조회하면 조회수가 3 증가한다.
- **SC-005** (`FR-006`): 게시글 상세 조회 응답에 현재 조회수가 포함된다.
- **SC-006** (`FR-007`): 인기 게시글 목록은 좋아요 수 내림차순으로 정렬되어 최대 10건 반환된다.
- **SC-007** (`NFR-002`): 캐시 연결이 불가능한 상태에서도 게시글 상세 조회 요청은 정상적으로 200 응답을 반환한다.
- **SC-008** (`NFR-001`): 동일 게시글에 다수의 동시 좋아요 요청이 발생해도(예: 서로 다른 사용자 100명이 동시에 좋아요) 최종 좋아요 수가 정확히 일치한다.
- **SC-009** (`NFR-005`): 동일 게시글을 캐시 미스 상태에서 조회한 뒤 곧바로 다시 조회하면, 두 번째 조회는 영속 저장소(MongoDB)에 접근하지 않고 캐시로부터 응답한다.

## 범위 외 (Out of Scope)

- 댓글(Comment) 좋아요 — 이번 spec은 게시글(Post) 좋아요만 다룬다.
- 조회수 어뷰징 방지(동일 사용자·세션의 짧은 시간 내 중복 조회 차단, 봇 트래픽 필터링) — 향후 spec에서 다룬다.
- 좋아요 알림·피드 — 다루지 않는다.
- 캐시 사전 워밍(warm-up), 캐시 예열 전략 — 다루지 않는다.
- 인기 게시글 목록의 페이지네이션 — 상위 10건 고정 노출만 지원한다.
- 강의(Course) 좋아요 — 이번 spec의 범위는 커뮤니티 게시글(002)에 한정한다.

## 미결 사항 (Open Questions)

없음. 좋아요 대상(게시글만)·토글 정책(재클릭 시 취소)·조회수 중복 방지 여부(미지원)·인기 게시글 정렬 기준(좋아요 수)·노출 개수(상위 10개)는 사용자와의 대화로 확정하였다.
