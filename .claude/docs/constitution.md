# Project Constitution

> 이 문서는 프로젝트의 **불변 원칙**을 정의한다.
> 모든 설계·구현·검증 작업 전에 반드시 읽는다.
> constitution 조항이 전역 규칙과 충돌하는 경우, constitution이 우선한다.
>
> - **변경 원칙**: 조항은 팀 전체의 합의 없이 변경하지 않는다. (1인 연습 프로젝트이므로 사용자 본인의 명시적 결정으로 변경)
> - **기준 커밋**: 이 문서의 내용은 **§3 변경 이력의 마지막 commit 기준**이다. 최초 작성 시점에는 아직 커밋 이력이 없다.

## 1. 프로젝트 원칙

### P-001. 도메인 순수성 원칙

`domain` 패키지는 Spring·JPA 등 특정 프레임워크에 의존하지 않는 순수 Kotlin으로 작성한다.

- 도메인 엔티티·값 객체·도메인 서비스에 `@Entity`, `@Autowired` 등 프레임워크 애노테이션을 붙이지 않는다.
- 영속성 매핑은 `infrastructure` 계층의 별도 클래스(JPA Entity 등)가 담당하고, 도메인 객체와 상호 변환한다.
- 이 원칙은 월부닷컴 채용공고의 "DDD, clean architecture 지향 리팩토링" 요구사항을 연습하기 위한 이 프로젝트의 핵심 학습 목표이므로 예외를 최소화한다.

**Gates 질문**: 이 구현이 도메인 계층에 프레임워크 의존성을 주입하지 않는가?

---

### P-002. 성능 원칙 (조정)

이 프로젝트는 학습·포트폴리오 목적이라 실제 트래픽 부하는 없지만, 채용공고의 "4,100만 규모 대용량 콘텐츠 트래픽" 대응 역량을 연습으로 보여주는 것이 목적이므로, 명백히 알려진 비효율(N+1 쿼리, 불필요한 동기 블로킹 I/O 등)은 인지한 상태에서 방치하지 않는다.

**Gates 질문**: 이 구현이 알려진 비효율(N+1 등)을 새로 도입하지 않는가?

---

### P-003. 호환성 원칙 (조정)

외부 소비자가 없는 연습 프로젝트이므로 완전한 하위 호환 유지 의무는 없다. 다만 같은 spec 내에서 이미 구현·테스트된 API 계약(요청/응답 스키마)을 같은 spec 작업 중 임의로 변경하지 않는다 — 설계(`plan.md`)와 구현의 정합성을 유지하기 위함이다.

**Gates 질문**: 인터페이스 변경이 같은 spec 내 이미 합의된 계약과 어긋나지 않는가?

---

### P-004. 테스트 원칙

모든 기능 요구사항(FR-XXX)은 검증 시나리오(SC-XXX)와 대응하는 테스트 코드를 갖춰야 한다.

- spec.md에 수용 기준(SC-XXX)이 없는 기능 요구사항은 구현을 시작하지 않는다.
- 동시성 등 단위 테스트로 검증 불가능한 항목은 통합 테스트로 커버한다 (예: 001 spec의 SC-006).

**Gates 질문**: 모든 FR-XXX에 대한 검증 시나리오가 spec.md에 정의되어 있는가?

---

### P-005. 스펙 범위 원칙

구현은 spec.md에 명시된 범위를 벗어나지 않는다.

- spec 외 리팩토링, 기능 추가, 성능 개선은 별도 spec으로 분리한다.
- 범위 외 변경이 불가피한 경우 plan.md에 근거를 명시하고 spec.md를 먼저 업데이트한다.

**Gates 질문**: 이 구현이 spec.md 범위를 벗어나는 변경을 포함하지 않는가?

---

## 2. 예외 처리 방식

constitution 위반이 불가피한 경우 아래 절차를 따른다.

1. `plan.md`의 **"예외 사항"** 항목에 위반 조항과 근거를 명시한다.
2. 대안(위반 범위 최소화, 단계적 해소 계획 등)을 함께 기재한다.
3. 사용자의 확인을 받은 후 구현을 진행한다.
